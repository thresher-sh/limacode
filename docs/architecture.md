# Architecture

Limacode is a multi-file bash CLI that generates Lima VM configurations on the fly and manages the VM lifecycle around AI coding agent sessions.

## Project Structure

```
limacode.sh                  # Main CLI -- sources lib/*, dispatches subcommands
lib/
  config.sh                  # Read/write ~/.limacode/config
  registry.sh                # Load and validate agent definitions
  yaml.sh                    # Generate Lima YAML configs
  vm.sh                      # VM lifecycle (create, start, stop, shell, delete)
  network.sh                 # iptables rule generation for --restrict-dns
  image.sh                   # Base image build and update
registry/
  claude-code.sh             # Agent definition: Claude Code
  opencode.sh                # Agent definition: OpenCode
  pi.sh                      # Agent definition: Pi.dev
scripts/
  provision.sh               # Base image provisioning (installs all agents)
  install.sh                 # curl | sh interactive installer
  iptables.sh                # Placeholder (rules generated by lib/network.sh)
tests/
  limacode.bats              # CLI integration tests
  lib/*.bats                 # Per-module unit tests
  fixtures/                  # Test data (sample agents, configs)
docs/                        # Documentation
```

## Module Responsibilities

Each `lib/*.sh` file has a single responsibility and exports functions with a namespace prefix.

### config.sh

Reads and writes `~/.limacode/config`, a flat `KEY=value` file. Provides defaults for all config keys. The precedence order is:

```
CLI flags  >  config file  >  built-in defaults
```

**Functions:** `config_get`, `config_set`, `config_list`

### registry.sh

Loads agent definitions from `registry/*.sh` files. Validates each file before sourcing it into the shell. Validation includes:

- Static checks (disallowed patterns, required variables and functions)
- Post-source checks in a subshell (name matches filename, functions callable, name format)
- ShellCheck enforcement in CI (not at runtime)

**Functions:** `registry_validate`, `registry_load`, `registry_find`, `registry_list_agents`

### yaml.sh

Generates Lima YAML configuration dynamically based on the user's options. Handles:

- Platform detection (VZ on macOS, QEMU on Linux)
- Mount type selection (virtiofs on macOS, 9p on Linux)
- Primary project mount at `~/workspace/current`
- Additional `--adir` mounts at `~/workspace/<name>`
- Network restriction provisioning (delegates to `network.sh`)

Environment variables are intentionally **not** written to the YAML file. They are injected at runtime via `--preserve-env` to avoid writing secrets to disk.

**Functions:** `yaml_generate`, `_yaml_detect_platform`, `_yaml_detect_mount_type`

### vm.sh

Wraps `limactl` commands. Handles instance naming, lifecycle management, and interactive instance selection.

Instance names follow the format `limacode-<agent>-<cwd-hash>-<unique-int>`:
- `cwd-hash`: first 6 characters of the SHA-256 of the absolute working directory path
- `unique-int`: incremented for parallel sessions against the same directory

**Functions:** `vm_instance_name`, `vm_cwd_hash`, `vm_create`, `vm_start`, `vm_shell`, `vm_shell_with_env`, `vm_stop`, `vm_delete`, `vm_cleanup`, `vm_list`, `vm_pick_instance`

### network.sh

Generates iptables scripts for `--restrict-dns`. The generated script:

1. Flushes existing OUTPUT rules
2. Allows loopback, established connections, Lima private ranges, DNS
3. Resolves each allowed domain and permits its IPs
4. Sets OUTPUT policy to DROP
5. Installs a cron job to re-resolve IPs every 5 minutes

**Functions:** `network_generate_iptables_script`

### image.sh

Manages the pre-built base VM image. `image_build` creates a temporary VM, runs `scripts/provision.sh` inside it, exports the disk image to `~/.limacode/images/`. `image_update` is an alias that rebuilds the image with latest agent versions.

**Functions:** `image_local_path`, `image_exists`, `image_checksum_cmd`, `image_build`, `image_update`

## Data Flow

```
User runs: limacode --agent opencode --env API_KEY=xxx

  1. limacode.sh parses flags, merges with config defaults
  2. registry.sh validates and loads registry/opencode.sh
  3. yaml.sh generates Lima YAML (mounts, platform, image)
  4. vm.sh creates and starts the Lima VM via limactl
  5. vm.sh exports env vars in a subshell, runs agent via limactl shell --preserve-env
  6. Agent runs inside the VM against ~/workspace/current
  7. On exit, vm.sh stops and deletes the VM (trap handler)
```

## Platform Behavior

| | macOS | Linux |
|---|---|---|
| VM backend | VZ (Apple Virtualization.framework) | QEMU (with KVM if available) |
| Mount type | virtiofs (near-native speed) | 9p |
| Prerequisites | Lima (via Homebrew) | Lima + QEMU |
| Min OS version | macOS 13.5 | Kernel 5.x+ |

## Secret Handling

Secrets (API keys, tokens) follow a strict path:

1. User provides via `--env KEY=VALUE` or `limacode config env`
2. Limacode exports them in a **subshell** before calling `limactl shell --preserve-env`
3. Lima forwards the subshell's environment into the guest
4. Secrets are **never written** to the Lima YAML file on disk

The subshell scoping ensures env vars don't leak into the user's parent shell session.

## Testing

All tests use [BATS](https://bats-core.readthedocs.io/) (Bash Automated Testing System).

- **Unit tests** (`tests/lib/*.bats`): test each module in isolation, no VM required, fast
- **Integration tests**: gated behind `LIMACODE_INTEGRATION=1`, require a running Lima installation
- **CI**: ShellCheck linting + BATS tests on both macOS and Ubuntu (see [CI/CD](cicd.md))

Test helpers provide temp directory setup/teardown and stub `log`/`error` functions that lib modules depend on.
